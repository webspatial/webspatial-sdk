Validation harnesses + documentation rollout for `openspec/lazy-load-spatial-runtime` (PR 6 of 6). No public API change beyond what PRs 1–5 already shipped; this PR closes the OpenSpec change with the §10 (docs) + §13 (SSR / hydration) + §14 (stateless utility) + §15 (facade ↔ real-impl parity) tasks.

**Documentation (`tasks.md §10`)**:

- `packages/react/README.md` — adds the "Web-first, spatial-enhanced" section (boot recipe, default-fallbacks-per-component table linked back to `spec.md` "Component facades" Requirement, custom-wrapper escape hatch using `useSpatialReady`), the React 18+ peer-dep note, the Bundler-compatibility section linked to the migration guide's tested-targets list, and the "Stateless utility APIs" cross-reference table that points back to `spec.md` "Stateless utility APIs and pure re-exports remain in the default entry" Requirement (per `tasks.md §14.8`).
- `docs/migration/lazy-load-spatial-runtime.md` — full migration guide with TL;DR + step-by-step recipes ("Drop `@webspatial/vite-plugin`", "Call `bootSpatial()` before your first render", "Use HOC facades instead of internal containers", "Drop `/web` and `/default` subpaths", "`createElement` deprecation"), the boot-before-vs-after-`hydrateRoot` SSR sub-case discussion, the bundler-capability matrix + tested-target list, the out-of-scope-environments list (Module Federation, Next.js Turbopack, Webpack 4, CommonJS-only pipelines), the RSC consumer note, the `useMetrics` per-instance pinning behavioral change, and the BREAKING-changes summary.

**Build-pipeline change (`tasks.md §13.1`)**:

- `packages/react/tsup.config.ts` — adds an `onSuccess` hook that injects `'use client';` at the top of `dist/index.js` after every build. Reasoning is documented inline: tsup's `splitting: true` (per `tasks.md §8.1`) merges multiple source files into shared chunks, which prevents esbuild from preserving a per-source-file `'use client'` directive at chunk boundaries. The pragmatic answer is to inject the directive at the public RSC boundary (`dist/index.js`) — the only file consumer-side bundlers see directly. Internal chunks (`dist/chunk-*.js`) and the dynamic-import target (`dist/spatial.js`) are reached transitively after the consumer's bundler has already crossed the client boundary, so they do NOT need their own directive (and `dist/jsx/jsx-runtime.js` / `dist/jsx-dev-runtime.js` MUST NOT carry it because they do not call React hooks).

**SSR / hydration validation (`tasks.md §13`)**:

- `packages/react/src/__tests__/use-client-directive.test.ts` (`§13.1`) — asserts `dist/index.js` begins with `'use client';` and that `dist/jsx/jsx-runtime.js`, `dist/jsx/jsx-dev-runtime.js`, `dist/spatial.js` do NOT carry the directive.
- `packages/react/src/__tests__/ssr-hydration.test.tsx` (`§13.2 / §13.3 / §13.4 / §13.5 / §13.6`) — covers synchronous SSR via `renderToString` (no chunk fetch, fallback HTML, `onSpatialLoadError` listeners not invoked, `bootSpatial()` resolves synchronously, `useMetrics` placeholder returns documented constants), streaming SSR via `renderToPipeableStream` (parity with synchronous SSR, no Suspense boundary on the SDK side), `useSpatialReady` `getServerSnapshot` stability (referential identity across siblings + no subscription side-effects), and both hydration round-trip timings (boot AFTER hydrate; boot BEFORE hydrate). Both round-trip tests assert no React hydration-mismatch warning is logged and that the swap to real implementations happens on the React commit immediately after the bridge resolves.

**Stateless utility validation (`tasks.md §14`)**:

- `packages/react/src/__tests__/stateless-utilities.test.tsx` — covers `getAbsoluteUrl` (SSR no-window + relative-path resolution + malformed-input no-throw), `initScene` (pre-boot no-op + SSR-safe + Path 2 routing through `getSpatialImpl()?.getSession?.()`), `convertCoordinate` (returns input unchanged + no-throw with garbage refs), `enableDebugTool` (SSR-safe via `vi.doMock` of `core-sdk`'s module-cached `isSSREnv`, browser-mode attaches diagnostics, `inspectCurrentSpatialScene` throws a `bootSpatial()`-pointing message pre-boot and routes to the spatial impl post-boot), `WebSpatialRuntime.supports` (callable without boot, returns `false` for spatial keys in plain web and `true` under Puppeteer, no dynamic-import side effect), and Group C type-only re-exports (none of `SpatializedElementRef` / `EntityRef` / `ModelRef` / `ModelProps` / `CapabilityKey` / `RealityProps` / `SpatialTapEvent` are bound on the namespace object at runtime).

**Facade ↔ real-impl parity (`tasks.md §15`)**:

- `packages/react/src/__tests__/helpers/parity.tsx` (`§15.1 + §15.2`) — parity harness with `normalizeHtml` (sorts attribute order, collapses whitespace) + `renderStatic` (suppresses React's unknown-tag console.error noise from native `<model>`) + `captureBothPaths`. The contract is the two HTML strings MUST be structurally identical — caller asserts via `toBe`.
- `packages/react/src/__tests__/parity.test.tsx` (`§15.3 / §15.4 / §15.5`):
  - **Model parity**: Path 1 (facade fallback) and Path 2 (real `Model` with `enable-xr={false}`) produce structurally identical HTML and strip the same set of spatial-only event props. `displayName` matches across both.
  - **`useMetrics` parity**: Path 1 (placeholder constants) and Path 2 (real `useMetrics` under no-session conditions in `jsdom`) return numerically equivalent results for the documented assertion-grade values (`pointToPhysical(0)`, `pointToPhysical(1360)`, `physicalToPoint(1)`).
  - **Console-warning differential**: Path 1 in plain web is silent (no boot-was-forgotten log); Path 2 in WebSpatial runtime emits at most one console.warn per affected API per page lifetime.
- **`§15.8` follow-up tracking**: parity tests for `Reality`, the `*Entity` family (Box/Sphere/Cone/Cylinder/Plane/Model/AttachmentEntity), materials/textures/assets, `SceneGraph`/`World`, and the HOC wrappers are recorded as `it.todo` with explicit pointers to the OpenSpec follow-up. The current real-impl branches either (a) throw on missing `useRealityContext()` instead of rendering the documented `null` (Entity family / materials), (b) silently mount nothing instead of the documented `<div aria-hidden>` (Reality), or (c) lack a pinned Path 2 fallback in `runtime-capabilities` altogether (SceneGraph / HOCs). v1 ships with the parity harness in place; the spec amendment / real-impl alignment will land in v1.x without breaking consumers.
- **Drift-detection comments (`§15.6`)**: every facade source file (`Model.tsx`, `Reality.tsx`, `entities.tsx`, `resources.tsx`, `SceneGraph.tsx`, `withSpatialized2DElementContainer.tsx`, `withSpatialMonitor.tsx`) carries a `**PARITY (spec tasks.md §15.6)**` JSDoc block calling out the corresponding real-impl path and the parity test that enforces alignment, so future maintainers cannot quietly modify Path 1 without surfacing the drift.

**Build-output evidence**: rebuilding under the new pipeline measures `dist/index.js` gzipped at 1321 bytes (well under the 8192-byte SDK-side proxy budget; the `'use client';` injection added 13 bytes of headroom cost) and confirms `dist/spatial.js` continues to ship the spatial implementation as a separate chunk reached only via the bridge's dynamic `import()`.
