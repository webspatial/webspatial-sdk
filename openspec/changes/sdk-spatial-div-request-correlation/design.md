# Design: Refresh-safe frontend request correlation

## Context

SpatialDiv creation is asynchronous from the frontend perspective. The frontend opens a platform protocol URL and later receives a creation result associated with a request id. Incremental request ids are sufficient inside a single JavaScript execution context, but they are weak across page reload boundaries.

The SDK should make request identity globally unique enough for callback correlation without exposing implementation details to host platforms. Page lifecycle ownership should be represented separately as an epoch field.

## Goals / Non-Goals

**Goals:**

- Generate opaque `wsrid` values that are unique across page reloads in practical usage.
- Include `wsepoch` when the host runtime exposes a page lifecycle epoch to JavaScript.
- Keep `wsrid` and `wsepoch` as separate concepts: request identity vs. page ownership.
- Add timeout cleanup for pending request receivers.
- Preserve compatibility with existing protocol handling during migration.

**Non-Goals:**

- This change does not define how a host runtime rejects stale native objects.
- This change does not change the public `SpatialSession` or React APIs.
- This change does not make frontend unmount cleanup authoritative for page reloads.

## Decisions

### Decision 1: Use two protocol fields

`wsrid` is an opaque request id. It MUST be generated by the frontend SDK and MUST NOT require host platforms to parse its internal structure.

`wsepoch` is a comparable page lifecycle value. It MAY be injected by the host runtime into `window.__webspatialsdk__.pageEpoch` or an equivalent SDK-readable field. If absent, the SDK still emits `wsrid` and omits or leaves `wsepoch` empty.

Protocol example:

```text
webspatial://createSpatialized2DElement?wsrid=wsreq_k8f3a2_12&wsepoch=7
webspatial://createAttachment?wsrid=wsreq_k8f3a2_13&wsepoch=7
```

### Decision 2: Make `wsrid` opaque and refresh-safe

The SDK may build `wsrid` from a per-JS-context random nonce plus an incremental sequence. The resulting value is a single opaque protocol field. Host platforms consume it only as a correlation key.

### Decision 3: Keep compatibility with existing callback id paths

During migration, the platform adapter may also emit the legacy `rid` parameter or alias `rid` to `wsrid` so older native hosts can keep resolving pending requests.

The preferred callback key is `wsrid`. The legacy `rid` path should be removed only after all supported hosts consume the new fields.

### Decision 4: Add pending receiver timeout cleanup

Pending request receivers registered for protocol creation MUST be removed after a bounded timeout. Timeout should resolve or reject the request with a structured failure result instead of retaining the callback indefinitely.

### Decision 5: Keep the React cleanup contract unchanged

React unmount cleanup remains best-effort and continues to call `destroy()` for created spatial elements. Reload cleanup remains a host lifecycle responsibility and is not made authoritative by this frontend change.

## Risks / Trade-offs

- **[Risk] Hosts do not yet consume `wsepoch`** -> keep old behavior compatible and use logging/tests to verify metadata emission first.
- **[Risk] Timeout too short for slow host creation** -> choose a conservative default and allow internal adjustment without changing public APIs.
- **[Risk] Duplicate protocol params cause ambiguity** -> document `wsrid` as preferred and `rid` as compatibility-only.

## Migration Plan

1. Emit `wsrid` and `wsepoch` while preserving legacy `rid` compatibility.
2. Add tests for request id uniqueness across module reload simulation.
3. Add tests for timeout cleanup.
4. After all host platforms consume `wsrid`, remove legacy-only assumptions in a follow-up change.

## Open Questions

- What default timeout should be used for native content host creation?
- Should timeout behavior be configurable internally for automated tests?
