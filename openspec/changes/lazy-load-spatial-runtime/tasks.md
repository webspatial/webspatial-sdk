## 9. Size budget enforcement

There are two normative size measurements per the spec's "Default entry MUST NOT bundle spatial implementation" Requirement: an SDK-side proxy on `dist/index.js`, and a typical-case marginal-delta measurement on a downstream consumer fixture. §9 covers both, plus the structural identifier scan and the `sideEffects: false` declaration check from "Tree-shake friendliness".

- [x] 9.1 **SDK-side proxy** — add `packages/react/src/__tests__/size-budget.test.ts` (or similar location) that builds the package and asserts gzipped `dist/index.js` size is at most 8192 bytes (the SDK-side proxy from "SDK-side `dist/index.js` size proxy" Scenario). This runs in the SDK's own test suite without a fixture build
- [x] 9.2 **Marginal-delta fixture** — add a Vite-based consumer fixture (e.g. `tests/marginal-delta-vite/`) that builds two minimal apps via the same Vite ≥ 4 config: `app-base` (no SDK import) and `app-typical` (`import { Model, bootSpatial } from '@webspatial/react-sdk'`, plus minimal usage `<Model />` and `await bootSpatial()` so the imports survive tree-shaking). The test asserts gzipped marginal delta (`app-typical` minus `app-base`) is at most 8192 bytes (the product-level contract from "Marginal bundle delta on a typical consumer" Scenario). The fixture MUST use the SDK from `packages/react/dist` (not source) to reflect the published surface
  - **Status**: PR 5 originally shipped this fixture with a temporary `REGRESSION_GUARD_BYTES = 24576` ceiling because the measured delta was ~19 KB (`@webspatial/core-sdk`'s tsup config used `bundle: true, splitting: false`, collapsing the entire SDK into one giant bundle whose internal `__export` runtime helper for wildcard re-exports defeated downstream tree-shaking). ## 15. Facade ↔ real-impl unsupported parity validation

After lazy-load, every Group A public component / hook has TWO independent fallback code paths that MUST produce structurally identical observable behavior:

- **Path 1 — boot-bundle facade fallback** (lives in default entry): exercised when `isSpatialReady() === false` for any reason (non-WebSpatial browser, SSR, boot in flight, boot rejected, boot never called). Pinned by `spec.md` "Component facades" / "Hook placeholders" Requirements.
- **Path 2 — spatial-chunk real-impl unsupported fallback** (lives in `@webspatial/react-sdk/spatial`): exercised when `bootSpatial()` HAS resolved AND the real implementation is mounted, but the underlying capability check returns `false` (e.g. shell version too old, capability not in `core-sdk` capability table). Pinned by `runtime-capabilities` "Unsupported behavior contracts" Requirement.

These two paths are physically separate code: the facade lives in PR 2's `packages/react/src/facades/*.tsx`; the real-impl unsupported branch lives in pre-existing components such as `packages/react/src/Model.tsx` / `packages/react/src/reality/components/Reality.tsx`. **Without explicit parity tests, the two paths can silently drift** — e.g. someone changes the facade's `<div aria-hidden>` to `<span>` for `Reality`'s fallback but forgets to update the real-impl's `<div aria-hidden>` branch, or vice-versa.

§15 adds parity tests that catch this class of regression.

- [ ] 15.1 **Parity harness**: add a test utility (e.g. `packages/react/src/__tests__/helpers/parity.ts`) that, given a `componentName: 'Model' | 'Reality' | 'BoxEntity' | ...` and a `props` shape, mounts the facade in two contexts and returns the rendered HTML strings:
  - **Context 1** (Path 1): non-WebSpatial UA stub (`navigator.userAgent` = plain Chrome), `bootSpatial()` not called, `isSpatialReady() === false` → boot-bundle facade fallback fires
  - **Context 2** (Path 2): WebSpatial UA stub (e.g. `Puppeteer` UA so `detectSpatialRuntime() === 'puppeteer'`), `await bootSpatial()` resolved with the real spatial implementation loaded under mocked runtime / capability state (not an empty placeholder namespace), and `vi.spyOn(WebSpatialRuntime, 'supports').mockReturnValue(false)` for the relevant capability key → the real-impl unsupported branch fires
- [ ] 15.2 **Parity assertion contract**: the harness asserts the two HTML strings are **structurally identical** (same tag, same attribute set ignoring React-internal data-attrs, same children structure). Style attributes are compared as parsed `style` objects to avoid false negatives from key order. Refs forwarded in both contexts MUST behave identically (both `null` for `null`-fallback components; both attached to the host element for `Model` / `Reality`)
- [ ] 15.3 **Parity test cases per facade table** — one parametrized test per Group A facade with documented Scenario 2 fallback:

  - `Reality` (Path 1: `Reality fallback preserves layout` Scenario; Path 2: `runtime-capabilities` "`Reality` unsupported fallback" Scenario)
  - `Entity`, `BoxEntity`, `SphereEntity`, `ConeEntity`, `CylinderEntity`, `PlaneEntity`, `ModelEntity`, `AttachmentEntity` (Path 1: facade table `null`; Path 2: `runtime-capabilities` "Unsupported HTML component rendering" Scenario)
  - `UnlitMaterial`, `Material`, `Texture`, `ModelAsset`, `AttachmentAsset` (same pattern as entities)
  - `SceneGraph` / `World` (Path 1: facade table `<>{children}</>`; Path 2: **currently NOT pinned by `runtime-capabilities`** — the real-impl fallback for `supports('SceneGraph') === false` is implementer's choice today; this test SHOULD assert the real-impl matches the facade's transparent-children behavior, and the implementer MUST either adjust the real-impl or push back via a spec amendment)
