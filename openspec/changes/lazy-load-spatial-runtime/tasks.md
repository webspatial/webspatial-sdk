## 15. Facade ↔ real-impl unsupported parity validation

After lazy-load, every Group A public component / hook has TWO independent fallback code paths that MUST produce structurally identical observable behavior:

- **Path 1 — boot-bundle facade fallback** (lives in default entry): exercised when `isSpatialReady() === false` for any reason (non-WebSpatial browser, SSR, boot in flight, boot rejected, boot never called). Pinned by `spec.md` "Component facades" / "Hook placeholders" Requirements.
- **Path 2 — spatial-chunk real-impl unsupported fallback** (lives in `@webspatial/react-sdk/spatial`): exercised when `bootSpatial()` HAS resolved AND the real implementation is mounted, but the underlying capability check returns `false` (e.g. shell version too old, capability not in `core-sdk` capability table). Pinned by `runtime-capabilities` "Unsupported behavior contracts" Requirement.

These two paths are physically separate code: the facade lives in PR 2's `packages/react/src/facades/*.tsx`; the real-impl unsupported branch lives in pre-existing components such as `packages/react/src/Model.tsx` / `packages/react/src/reality/components/Reality.tsx`. **Without explicit parity tests, the two paths can silently drift** — e.g. someone changes the facade's `<div aria-hidden>` to `<span>` for `Reality`'s fallback but forgets to update the real-impl's `<div aria-hidden>` branch, or vice-versa.

§15 adds parity tests that catch this class of regression.

- [x] 15.1 **Parity harness**: add a test utility (e.g. `packages/react/src/__tests__/helpers/parity.ts`) that, given a `componentName: 'Model' | 'Reality' | 'BoxEntity' | ...` and a `props` shape, mounts the facade in two contexts and returns the rendered HTML strings:
  - **Context 1** (Path 1): non-WebSpatial UA stub (`navigator.userAgent` = plain Chrome), `bootSpatial()` not called, `isSpatialReady() === false` → boot-bundle facade fallback fires
  - **Context 2** (Path 2): WebSpatial UA stub (e.g. `Puppeteer` UA so `detectSpatialRuntime() === 'puppeteer'`), `await bootSpatial()` resolved with the real spatial implementation loaded under mocked runtime / capability state (not an empty placeholder namespace), and `vi.spyOn(WebSpatialRuntime, 'supports').mockReturnValue(false)` for the relevant capability key → the real-impl unsupported branch fires

The harness lives in `helpers/` so it can be reused by future parity tests as additional facades' Path 2 contracts get pinned (see §15.8). The actual Path 1 / Path 2 plumbing (UA stubs, bridge mocks, `bootSpatial()` ordering) lives in the test files that call the harness — `parity.test.tsx` keeps the per-test setup explicit so each Path 2 mock is auditable inline.

## 16. Eager-mode entry for spatial-only consumers

This section implements the "Eager-mode entry for spatial-only consumers" Requirement (in `specs/spatial-lazy-load/spec.md`). The eager entry is an additive sibling to the lazy-load default entry — it shares the same TypeScript surface but statically links the spatial implementation so spatial-only consumers pay one network request instead of two and skip the `bootSpatial()` round-trip. It also implements the "Two distribution forms share packaging hygiene" Requirement by re-exporting Group B / Group C utilities by reference from the default entry (not redeclaring) so behavior parity is byte-identical.

- [x] 16.1 Add `packages/react/src/runtime/eagerStubs.ts`. Exports compatibility stubs that the eager entry uses to satisfy the lazy-load runtime API surface:
  - `bootSpatialEager(): Promise<void>` — returns `Promise.resolve()`. In dev builds emits a one-shot `console.warn` reminding the user that `bootSpatial()` is unnecessary on the eager entry; the warning MUST be guarded by a module-level `let warned = false` so it never fires twice in the same page lifetime. In production builds the warning code MUST be tree-shake-friendly (gated on `process.env.NODE_ENV !== 'production'` or equivalent).
  - `isSpatialReadyEager(): boolean` — returns `true` synchronously.
  - `useSpatialReadyEager(): boolean` — returns `true` on first and every subsequent render. Implementation does NOT need `useSyncExternalStore` since the value is constant; a plain `() => true` is acceptable as long as it is a stable module-level reference (so consumer React doesn't re-subscribe per render).
  - `onSpatialLoadErrorEager(_cb: (err: WebSpatialBootError) => void): () => void` — returns a no-op unsubscribe function. The callback MUST never be invoked.
