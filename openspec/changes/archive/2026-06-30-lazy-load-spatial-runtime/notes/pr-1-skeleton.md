## Section 1 — Overview

PR 1 introduces five runtime modules under `packages/react/src/runtime/`, with dependency direction:

- `detect.ts` → `@webspatial/core-sdk` only
- `errors.ts` standalone
- `bridge.ts` → `errors.ts` + dynamic `import('@webspatial/react-sdk/spatial')` (no React)
- `boot.ts` → `detect.ts` + `bridge.ts`
- `useSpatialReady.ts` → `react` + `detect.ts` + `bridge.ts`

The bridge owns the spatial chunk load, readiness bit, and error listeners; boot is the single user-facing activation entry that gates work on `detectSpatialRuntime()`; `useSpatialReady` is the client hook that subscribes to bridge readiness with a plain-web/SSR fast path.

This artifact defines **TypeScript types, function/class signatures, module-level state declarations, and JSDoc only**. Every function body is a deliberate placeholder so implementers can treat it as a contract handshake; **no business logic** belongs in this draft.

In addition to the five runtime files, PR 1 also creates `packages/react/src/spatial/index.ts` per `tasks.md §3.2` — a thin bridge-facing namespace that re-exports from the existing implementation source paths (do NOT move files; that is `§3.1`, scheduled for PR 4). This file is not detailed below; see `tasks.md §3.2` and the "Phasing note" within it for the rules.

## Section 4 — Open questions / spec gaps surfaced

1. **`__internalBootPromise` in `boot.ts`** — `tasks.md §2.3` does not require extra boot-level promise state if `loadSpatialImpl` fully owns idempotency; leaving this as an implementer choice (`TODO` in §2.4.2).

2. **`WebSpatialBootError` `message` string** — spec mandates `name`, `cause`, `attempt` only; default message text is **not** normative.

3. **Failure listener ordering with `Set` vs array** — spatial-lazy-load `bootSpatial` Requirement says callbacks run in **registration order**; ECMAScript `Set` iteration follows insertion order for `Set<function>`, but the spec also says "stored in a Set" in `design.md` Decision 2 — **compatible**, but if reviewers want belt-and-suspenders, an explicit queue could be clearer (not required by spec).

4. **`detectSpatialRuntime()` stability vs `useState` initializer** — `design.md` Decision 7 says detection is treated stable for the page lifetime; `useSpatialReady` uses one-shot detection in `useState(() => …)`. **(none blocking)**; only relevant if tests ever mutate UA mid-page.

5. **Full hydration / "swap after hydration commits"** — contract is spread across `useSpatialReady` + React internals; **unit** tests in `useSpatialReady.test.ts` cannot fully prove "Switch to spatial happens after hydration commits" without `hydrateRoot` integration (`tasks.md §13.5–§13.6`). Not a signature gap — flag for **test placement**, not API shape.

**Otherwise:** all exported signatures above are grounded in `specs/spatial-lazy-load/spec.md`, `specs/runtime-capabilities/spec.md` (delta), `design.md` Decisions 2–3 and 7, and `tasks.md §2.1–§2.6`, §4.1, §4.10.
